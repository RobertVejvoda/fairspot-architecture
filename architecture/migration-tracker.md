# Architecture Migration Tracker

This tracker controls how legacy FairSpot documentation is migrated into the TOGAF-style architecture repository.

The goal is consistency first. Every architecture layer must be visible up front, even when the content is incomplete. Missing or intentionally deferred layers are recorded as placeholders instead of being hidden in older documentation.

## Migration Rules

- `architecture/` is the authoritative architecture repository structure.
- Legacy pages under `business-layer/`, `application-layer/`, `technology-layer/`, `security/`, `production/`, and delivery trackers remain source evidence until explicitly migrated or superseded.
- Migration is not a copy operation. Content should be restated into the target architecture viewpoint when needed.
- If content is missing, create or keep a placeholder and mark the status as `Placeholder`.
- If content exists but conflicts with the target architecture direction, record the gap in [Gap Analysis](/architecture/architecture-states/gap-analysis).
- Durable architecture decisions stay in [Versions and Decisions](https://robertvejvoda.github.io/fairspot/#/versions-and-decisions).
- Operational runbooks can remain in `production/` until the technology and operations repository structure is mature enough to absorb or reference them cleanly.

## Status Legend

| Status | Meaning |
| --- | --- |
| Migrated | Content has been restated in the architecture repository and legacy material is no longer the primary source. |
| Partial | Some content has been migrated, but legacy evidence is still needed. |
| Placeholder | The repository has an explicit page or section, but content is not yet complete. |
| Deferred | Known scope is intentionally postponed. Deferred scope must still be visible. |
| Source Evidence | Legacy content is useful input but not yet migrated. |
| Keep Product-Level | Page remains outside `architecture/` because it is product identity, evaluator entry, roadmap, decision history, or glossary material. |
| Keep Runbook | Page remains outside `architecture/` because it is an operational procedure, local setup guide, smoke test, or deployment runbook. |
| Retire After Migration | Page can disappear after its relevant content is restated in the architecture repository and links are updated. |

## Consolidation Guardrails

Do not delete or move legacy pages until this tracker records the disposition and the active documentation links are updated.

- Product-facing pages can remain outside the architecture repository when they explain FairSpot as a product rather than architecture.
- Architecture-significant content must be restated into the target repository instead of copied verbatim.
- Detailed implementation contracts can stay as source evidence until the corresponding architecture artifact has enough target-level content and traceability.
- Operational runbooks can remain in `production/` while the architecture repository links to the deployment profile, runtime, security, and evidence they support.
- Deprecated directions, such as Reporting-owned PostgreSQL read storage, must be marked obsolete or deferred instead of silently carried into target architecture.
- Retired pages should be removed only in a final cleanup slice after link validation confirms that no active page depends on them.
